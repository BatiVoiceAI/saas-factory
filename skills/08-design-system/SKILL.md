# SaaS Factory — Étape 8 : Design system & maquettes (rôle Designer)

Objectif : produire le **`DESIGN.md` (source de vérité visuelle)** + les **maquettes des écrans clés**, avec un **minimum d'interaction** : une seule décision est demandée à l'utilisateur (la **direction visuelle**), tout le reste est dérivé automatiquement.

## À lire d'abord (une fois)
`_shared/lessons.md`, `_shared/safety-rails.md`, **`_shared/design-doctrine.md` (OBLIGATOIRE avant toute proposition — §Processus de recherche de direction, §Points de départ (graines), leviers, §Direction motion, marqueurs de slop + convergence interdits, porte distinctiveness)** ; si présent, `skills/phase-2-product/references/conventions.md`. Au mouvement 3 : `_shared/landing-playbook.md` (structure + copy de la landing).

## Principe
**Pas de design bespoke, pas d'AI slop — et pas de convergence par template.** Une base **opinionated et buildable** (shadcn/ui + Tailwind), une esthétique **recherchée par projet** via le **§Processus de recherche de direction de `_shared/design-doctrine.md`** (brand audit → audit concurrentiel visuel / white space → moodboard de références réelles → direction distinctive dérivée : typo expressive, couleur dérivée du métier, layout signature, parti-pris motion) — **pas** « choisir une des 5 recettes » (les recettes sont des **graines de registre**, jamais la sortie). Des tokens qui **mappent directement la stack de la Phase 3** — sinon design et code divergent. Les **marqueurs de slop** (dont la **convergence**, marqueurs 21-23) sont des interdits binaires ; la **porte distinctiveness** (test anti-convergence + check comparatif + rationale par page + reduced-motion) fait loi.

**HARD GATE.** Ici : le système visuel + les maquettes. Pas de code produit (Phase 4), pas de choix d'architecture (Phase 3). Livrables : `DESIGN.md` + `design/mockups/`.

## Entrées / sortie (contrat)
- **Lit :** `product/product-spec.md` (écrans/features Must, workflow cœur), `research/positioning.md` (feel de marque, catégorie), `research/idea-brief.md` (type, cible), `product/pricing.md` (tiers/prix pour le copy de la landing), `_shared/design-doctrine.md` (recettes + leviers + interdits), `_shared/landing-playbook.md` (structure canonique + règles de copy).
- **Écrit :** `DESIGN.md` (racine, source de vérité, template `assets/templates/design-system.md` — inclut l'**artefact « direction »** (territoire nommé + white space + 2-3 références réelles + test anti-convergence), **1-2 éléments signature**, la **Direction motion** (parti-pris + runtime + reduced-motion) et le **Rationale par page** (une ligne « intention » par type de page)) + `design/mockups/` (écrans clés générés) + `design/brand-assets/` (visuels générés si besoin) + `.saas-factory/state.md` (état : étape 8 faite, direction retenue = territoire nommé + couleur de marque + motion signature, source charte = import/générée).

## Procédure normée (interaction minimale)
### 1. Une seule question : la DIRECTION (`AskUserQuestion`)
« As-tu déjà une charte graphique / un design system ? »
- **Oui → il dépose un dossier ou un fichier.** Lis-le et **extrais les tokens** dans `DESIGN.md` : couleurs, police(s), logo, spacing/radius s'ils y sont. Gère les formats courants (tokens JSON/CSS, config Tailwind, feuille de style, PDF de charte, ou simplement logo + liste de couleurs). Ne demande rien de plus.
- **Non → RECHERCHE la direction, puis propose 3 DIRECTIONS complètes, réellement codées (pattern shotgun).** D'abord le **§Processus de recherche de direction** de `_shared/design-doctrine.md` : **brand audit** (attributs de marque, `positioning.md`) → **audit concurrentiel visuel** (5-8 concurrents/voisins → palette/typo/densité/motion du secteur → **white space** libre) → **moodboard de références réelles** (Mobbin/Land-book/Awwwards — 2-3 URL qui incarnent le white space). Puis **3 directions distinctes**, chacune **seedée sur un registre différent** (graines §Points de départ, M2) mais **taillée** au white space (couleur **dérivée du métier**, typo = voix de marque, layout signature, parti-pris motion) — pas 3 palettes nues, pas 3 recettes servies brutes, **jamais deux du même registre**. Chacune devient une **VARIANTE CODÉE** : un **hero + 1 section clé** en code (React/HTML), **diversité forcée** (3 registres, 3 squelettes de hero différents), **rendue et screenshotée**, chacune passant la **porte distinctiveness** (test anti-convergence : ne pourrait pas servir un autre métier ; ne ressemble ni à une recette brute ni à un autre projet de l'usine). Le fondateur choisit en regardant du **RÉEL**. Il en retient **une**. C'est la **seule** décision design de toute l'étape — et la variante retenue sert de **graine** au mouvement 3.

Pourquoi rechercher avant de proposer : « choisir une des 5 recettes » produit la **convergence** (AgencyDesk ≈ coiffeur) ; occuper le **white space de CE secteur** force la divergence. Pourquoi limiter à une décision : la direction est le seul choix vraiment subjectif, et une direction recherchée embarche déjà polices + palette + layout + motion **cohérents et taillés** ; tout le reste se dérive proprement. Pourquoi coder les 3 variantes : un aperçu abstrait fait choisir un fantasme, un rendu réel fait choisir un produit — et le code de la variante retenue est réutilisable, pas jeté. Détail des deux couloirs (import vs génération) : `references/procedure-detaillee.md` (mouvement 1) ; routage de format d'import : `references/matrices-decision.md` (M1) ; recherche + composition des 3 directions : `matrices-decision.md` (M2 + M3) ; recette de la question : `references/forcing-questions.md` (Q1).

### 2. Dériver tout le reste — automatiquement, via `frontend-design` (zéro question)
Une fois la direction fixée, lance **`frontend-design` une seule fois** pour dériver le système complet en **imposant les leviers de `design-doctrine.md`** : **neutres teintés** (jamais gris purs — hue partagée avec la marque), **paire typographique** = display à caractère (H1-H3, **voix de marque**) + sans neutre pour le corps, exposée en `--font-display`/`--font-sans`/`--font-mono`, **couleur de marque dérivée du territoire du métier** (jamais la teinte d'une recette recopiée), **bordures 1px > ombres** (une seule ombre tokenisée pour les éléments flottants), **échelles explicites** de radius (personnalité déclarée, déclinée depuis `--radius`) et de spacing (base-4 stricte), **layout signature** (parti-pris spatial du moodboard), **dark mode cohérent** (les neutres gardent leur teinte), et le **système de composants** (**shadcn/ui + Tailwind + icônes Lucide**). Matérialise dans `DESIGN.md` l'**artefact « direction »** (territoire nommé + white space + 2-3 références réelles + test anti-convergence) et la **Direction motion** (parti-pris : où/pourquoi/runtime/reduced-motion — §Direction motion de la doctrine). Les **marqueurs d'AI slop** (`design-doctrine.md` §Marqueurs — indigo/violet, gradient bleu→violet, Inter seule, gris purs, glassmorphism, ombres floues partout, **convergence 21-23**…) sont **INTERDITS** : un livrable qui en coche un est refusé. Intègre le **contraste a11y** dès les tokens (ratios texte/fond conformes) et **passe la porte distinctiveness** (test anti-convergence + check comparatif vs autres projets de l'usine). Écris le tout dans `DESIGN.md`. Pourquoi une seule passe après le choix de direction : la recette contraint le reste (contraste, hiérarchie) ; la fixer d'abord évite les allers-retours. Ordre de dérivation + mode dégradé : `references/procedure-detaillee.md` (mouvement 2) ; le contrat « buildable » (mapping token→Tailwind/shadcn) et la procédure de contraste AA (light **et** dark) : `references/tokens-a11y.md` ; dark mode oui/non : `matrices-decision.md` (M6).

### 3. Générer les écrans clés du PRD
On part du principe que l'utilisateur **n'a pas de maquette** → on les **génère**. Dérive la liste des écrans depuis le PRD (features **Must** + **landing** + auth + dashboard, **workflow cœur d'abord**) et génère-les avec **`frontend-design` en code (React/HTML production)** — fiable, **autonome**, sans dépendre d'un outil navigateur. ⚠️ **La landing suit `_shared/landing-playbook.md`** : structure canonique (11 sections, ordre imposé, testimonials **conditionnels** — rendus uniquement si l'intake fournit des témoignages réels) + **copy spécifique** dérivé de `research/positioning.md` + `product/pricing.md` + PRD (formules de headline, features orientées jobs, prix réels affichés) — **jamais un hero nu, zéro lorem/placeholder, zéro testimonial inventé**. Le visuel hero = **screenshot/mockup réel** de l'app (l'écran le plus parlant du mouvement 3), pas une illustration abstraite. Mobilise l'**arsenal créatif** (`design-doctrine.md` §Arsenal) : pars de la **variante shotgun retenue** (graine), **duplique** les structures éprouvées (blocs vendorables, squelettes de sections) puis **re-thème** systématiquement (tokens + polices + radius + contenu — au point de ne plus ressembler à son origine), animations **selon la Direction motion** de `DESIGN.md` (Motion pour l'état/presence ; dotLottie pour le décoratif one-shot ; Rive pour l'interactif piloté par l'état ; GSAP seulement pour une scène scroll orchestrée — via `<MotionAsset>`, lazy, theming-aware, **`prefers-reduced-motion` respecté**) sous les **règles d'animation anti-slop** de l'arsenal, og-image de la landing **générée en code** (`next/og`, tokens du thème). **Renseigne le Rationale par page** dans `DESIGN.md` : une ligne « intention » par type de page (pourquoi ce layout / cette hiérarchie / cette anim servent le job de CETTE page) — et **le rendu doit tenir ce rationale** (la QA design 13/14 le vérifie). Écris dans `design/mockups/`. Pourquoi du code plutôt qu'un outil de maquette : autonome, respecte exactement le `DESIGN.md`, **réutilisable directement** en Phase 3. Dérivation ordonnée de la liste d'écrans : `references/matrices-decision.md` (M4) ; sous-procédure + fidélité stricte au système : `references/procedure-detaillee.md` (mouvement 3).

### 4. Génération de visuels — Nano Banana Pro
Quand le design a **besoin d'imagerie** (hero éditorial/atmosphérique, **OG image**, illustration d'**empty-state**, spot art, favicon/wordmark), **GÉNÈRE** le visuel avec **Nano Banana Pro** (`gemini-3-pro-image`, image Google/Gemini) via le helper **`_shared/blocks/web-saas/scripts/generate-visual.mjs`** — **prompt dérivé de `DESIGN.md`** (palette de marque, typo, métier, parti-pris), **jamais du stock générique ni un placeholder** (c'est le levier d'anti-convergence côté image : deux métiers → deux prompts → deux visuels). Sortie **`public/generated/`**, référencée via **`next/image`** ; l'**OG image** est **générée** et branchée dans les `metadata` (`openGraph.images`) ; **favicon/wordmark** possibles. ⚠️ **Jamais d'image IA maquillée en screenshot produit** (hero « voici l'app », « comment ça marche ») : interdit doctrine n°17 — la preuve produit reste un **vrai screenshot** du mouvement 3 ; jamais de fausse preuve sociale (n°18). La **clé `GEMINI_API_KEY`** est déposée au setup (infra-setup) — lue en **variable d'environnement**, **jamais en dur**. Décider quoi générer : `references/matrices-decision.md` (M5) ; discipline de génération + repli honnête si la clé manque : `references/procedure-detaillee.md` (mouvement 4). *(En Phase 2, les visuels exploratoires peuvent aller dans `design/brand-assets/` ; les visuels **produit livrés** sont générés/câblés au **build** — `public/generated/`, étape 12, `skills/12-build/references/walking-skeleton.md`.)*

## Garde-fous
- **Buildable ET appliqué, pas décoratif.** Le `DESIGN.md` déclare le **mapping exact** vers `app/globals.css` (bloc `:root` + `.dark`, valeurs HSL/OKLCH prêtes). ⚠️ **La Phase 4 DOIT l'appliquer** — écrire ces tokens dans `app/globals.css` en **écrasant** les défauts du châssis. Un `DESIGN.md` qui reste un document mort (le produit sort au look **générique noir/blanc du châssis**) = **échec**, pas un design. La charte n'est « faite » qu'une fois **dans le code**.
- **Zéro marqueur d'AI slop.** La checklist de `_shared/design-doctrine.md` fait loi : un design (tokens, mockup ou landing) qui coche **un** marqueur (primary indigo/violet, gradient bleu→violet, Inter seule, gris purs, hero centré générique, 3 cartes identiques, testimonials inventés…) est **REFUSÉ avant la porte** — retour au mouvement concerné, pas de rustine ponctuelle.
- **Zéro convergence — porte distinctiveness.** La direction est **recherchée** (audit concurrentiel visuel / white space / moodboard), pas piochée dans 5 recettes. Elle doit passer : **(a)** test anti-convergence (ne pourrait pas servir un autre métier tel quel), **(b)** check comparatif (ne ressemble ni à une recette non modifiée, ni à un autre projet de l'usine), **(c)** rationale par page présent et tenu, **(d)** reduced-motion respecté. Un échec = retour recherche/dérivation, jamais la porte.
- **1-2 éléments signature + Rationale par page.** `DESIGN.md` déclare 1-2 éléments distinctifs assumés (ex. serif éditoriale en display dérivée de la marque, neutres chauds, layout asymétrique signature, section sombre de rupture) **et** une ligne « intention » par type de page (pourquoi ce layout/cette hiérarchie/cette anim servent le job de la page) — ce qui rend le produit reconnaissable et **justifié**, pas médian.
- **Direction motion assumée.** `DESIGN.md` déclare un parti-pris motion (motion signature + où/pourquoi + runtime Motion/dotLottie/Rive/GSAP + reduced-motion), câblé via `<MotionAsset>` — pas des micro-anims génériques.
- **Une seule décision utilisateur** (la direction). Tout le reste est dérivé.
- **A11y dès les tokens** (contraste, tailles de cible).
- **Cohérence > originalité.** Les écrans générés respectent le `DESIGN.md` (mêmes tokens, mêmes composants).
- **Ni Stitch ni Figma.** Import = simple dépôt de fichier/dossier ; génération = `frontend-design` (code) + Nano Banana (images). Plus autonome, plus simple.

Catalogue des modes d'échec (AI slop, design non buildable, divergence tokens↔mockups, contraste raté, trop de questions, secret committé…) et leur correction : `references/checklists-modes-echec.md`.

## Porte de sortie (validation charte)
Avant de clore, présente à l'utilisateur la **direction visuelle** retenue — le **territoire nommé + le white space visé** (vs les concurrents), la paire de polices, la palette dérivée, les **éléments signature**, la **Direction motion**, 1-2 maquettes clés, **et la checklist anti-slop passée** (les **19 points** de `design-doctrine.md`, dont la **porte distinctiveness** 16-19 : test anti-convergence, check comparatif, rationale par page, reduced-motion — résultat : 0 marqueur coché) — puis demande **validation** (`AskUserQuestion`). Un design qui coche un marqueur (slop **ou convergence**) **ne se présente pas à la porte** : il repart au mouvement concerné d'abord. S'il veut **ajuster la direction ou la dérivation**, reboucle sur le mouvement concerné (1 pour la recette, 2 pour la dérivation) puis **re-dérive** le reste — ne bricole pas un token isolé. C'est le seul point où la charte se valide côté utilisateur : une fois passée, elle est figée pour la Phase 3. DoD par artefact + checklist complète de la porte : `references/checklists-modes-echec.md`.

## Clôture d'étape
