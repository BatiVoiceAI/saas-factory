# SaaS Factory — Étape 12 : Build multi-agents (org CTO → Tech Lead → devs)

C'est le **niveau DEV** de ta cascade : on **construit** les features. Tu incarnes l'**org d'ingénierie** — le CTO-agent donne la direction, le Tech Lead-agent répartit à une **équipe de dev-agents** qui codent en parallèle. Tu ne codes pas toi-même : tu **orchestres** (`Task`).

**HARD GATE.** Ici on **construit et on auto-teste** (niveau dev). La **validation montante** (Tech Lead → CTO → Designer → CEO) = **étape 13**. Le **faux-client** = étape 14. Tu produis du **code + des tests verts par feature**, prêts à monter la cascade.

## À lire d'abord
`_shared/lessons.md`, `_shared/safety-rails.md`, `_shared/validation-cascade.md`, `_shared/stack-defaults.md`, `_shared/design-doctrine.md` (interdits design binaires — pour tout agent qui touche une surface UI), `_shared/landing-playbook.md` (structure + copy de la landing), `_shared/boucles-fermees.md` (chaque action de valeur ferme sa boucle : la passe d'intégration vérifie **zéro action muette**) ; le plan `tech/execution-plan.md` (étape 10 — items `[SÉCU]` + tâches d'intégration → registre de solde) ; le `CLAUDE.md` projet (étape 11). L'`archetype` (state.md : `web-saas`|`landing`|`automation`) **conditionne le SOCLE vérifié** à la sortie (web-saas **S1-S8** / landing **LP1-LP4** / automation **AU1-AU5** — `_shared/state-schema.md` §socle-par-archétype, SOURCE) ; le `type` (public|interne|perso, **orthogonal**) **adapte** la cérémonie (landing complète vs racine=login) sans choisir le socle : **route selon `skills/saas-factory/references/routing.md`** (ligne `12-build`, portes par archétype) — jamais recopier la matrice. **Moteur** : superpowers `subagent-driven-development` + `test-driven-development` + `using-git-worktrees` (`references/org-hierarchy.md`). Les **tests dev (TDD, recette)** alimentent le **livret de test** (`_shared/test-dossier.md` — tenu par le QA Analyst).

## Le moteur (on l'exécute, on ne le réinvente pas)
Le **pattern superpowers `subagent-driven-development`** est notre moteur **par-feature** (dispatch d'un dev-agent frais → implémente → self-review → la revue montante = étape 13). On y ajoute **notre org** (CTO / Tech Lead) et **notre fan-out multi-features**. TDD strict via `test-driven-development` ; isolation via `using-git-worktrees`. **Vendoré — exécute, ne réinvente pas** : `{PLUGIN_ROOT}/vendor/superpowers/skills/` (résolution §0 de `_shared/vendored-engine-protocol.md` ; liste + rôle de chaque moteur : `references/org-hierarchy.md`).

## Procédure
1. **Direction (CTO-agent)** — lis `tech/execution-plan.md` : walking skeleton, lanes, carte de délégation. Le **CTO** (`agents/cto.md`) fixe l'ordre et les contraintes techniques, spawn le Tech Lead. Rôles, org, routage de modèle : `references/org-hierarchy.md`.
2. **Walking skeleton d'abord** (séquentiel) — une tranche verticale mince qui **tourne** de bout en bout, mergée sur `main` avant tout fan-out ; dans le **même 1er geste** que la charte, il **retire la tuyauterie exemple du châssis** (entité CRUD d'exemple `items`, dashboard/sidebar d'exemple), **génère la vraie nav depuis le PRD**, et **génère les visuels on-brand** dont le design a besoin (`scripts/generate-visual.mjs`, Nano Banana Pro `gemini-3-pro-image`, prompts **dérivés de `DESIGN.md`** → `public/generated/`, câblés via `next/image` + OG dans `metadata` ; jamais du stock/placeholder ; repli honnête si `GEMINI_API_KEY`/`visuals` absents) — jamais le shell châssis dans le produit livré (`references/walking-skeleton.md`).
3. **Fan-out (Tech Lead-agent)** — pour les lanes parallèles : dispatche **un `feature-dev` par feature, dans son worktree** (`references/fan-out.md`), **en un seul message** ; anatomie exacte du dispatch dans `references/dispatch-brief.md`. Features liées → même lane coordonnée.
4. **Dev-loop (chaque dev-agent)** — code + **test technique (TDD)** + **gate de compilation** (`tsc` avec le **tsconfig STRICT du châssis**, `noUncheckedIndexedAccess` inclus → 0 erreur) + **recette** (ses critères d'acceptation du plan) + self-review ; **boucle jusqu'à ce que sa recette passe** (`references/dev-loop.md`). Écrit `status/<feature>.md`.
5. **Passe d'intégration** — après merge des lanes, le Tech Lead lance une **tâche d'intégration** (les bugs de jonction sont le coût du parallélisme, `_shared/lessons.md`) et une **porte de compilation = BUILD RÉEL de l'archétype** (`web-saas` : `next build` **+** `next lint`, pas seulement `tsc`) : un défaut qui passe `tsc` mais casse le build (ex. `export const` dans un fichier `"use server"`) n'est visible qu'au build réel. La passe vérifie **aussi** les garde-fous de sortie produit : (a) **charte appliquée** (`app/globals.css` ≠ défaut châssis) + **checklist anti-slop passée** (`_shared/design-doctrine.md`, binaire, desktop + mobile) + **visuels générés on-brand** (`public/generated/` via Nano Banana Pro, câblés `next/image`, dérivés de `DESIGN.md`, **0 stock/placeholder**) ; (b) **landing conforme au playbook** (`_shared/landing-playbook.md` : sections canoniques + copy spécifique, zéro placeholder/lorem) ; (c) **métadonnées brandées** (title/description/og — `lib/brand.ts` ≠ défaut châssis ; `grep "SaaS Factory Template"` sur le repo = 0) **+ favicon + OG image générée** on-brand (`openGraph.images` → `public/generated/og.png` ou route `next/og`) ; (d) **parcours premier usage complet** *(socle web-saas)* (signup OTP → mot de passe → onboarding qui crée l'entité cœur → dashboard non-vide) ; (e) **empty states avec CTA** sur chaque liste vide *(socle web-saas)* ; (f) **pages légales adaptées à la juridiction** (`jurisdiction`/`locale`, jamais « FR » en dur — un produit US/EN avec Terms + Privacy est conforme) présentes ; (g) **404 brandée + loading states** ; (h) **boucles fermées — zéro action de valeur muette** (`_shared/boucles-fermees.md` : chaque création/modification/annulation de l'entité cœur enfile+envoie sa notification à l'acteur ET à la contrepartie, dans la même transaction logique — jamais « TODO plus tard ») ; (i) **pricing = features livrées** *(public, P0.7)* : chaque ligne de pricing correspond à une **capacité effectivement livrée** (diff contre les Must implémentés du PRD) — feature absente = « bientôt » ou retirée, zéro vaporware public ; **(j) zéro tuyauterie exemple** *(universel, anti-scaffolding)* : aucune route/écran de l'entité exemple `items` atteignable, **zéro texte de dev rendu** (« à cloner », « entité exemple », « bloc CRUD », TODO, placeholder), **zéro lien de nav mort** (nav **dérivée des features du PRD**, `/billing` seulement si `billing=stripe`). Les garde-fous *(public)* — landing, pricing, légal — sont **calibrés par type** (interne/perso : pas de landing, racine=login, noindex ; `skills/saas-factory/references/routing.md`). **En plus** du calibrage par type, les items du **socle web-saas** — (d) premier usage / onboarding / entité cœur, (e) empty states, dashboard non-vide — sont **conditionnés par archétype** (axe orthogonal) : **n/a** pour un `landing` (pas d'auth/dashboard → on vérifie le socle **LP1-LP4** : sections landing-playbook + légal + waitlist/CTA + métadonnées/OG) et pour un `automation` (headless → socle **AU1-AU5** : config/secrets + historique runs/logs + healthcheck + boucle fermée + idempotence). Les garde-fous **vraiment universels** (charte, anti-slop, légal-juridiction, métadonnées/favicon, 404, zéro tuyauterie exemple) restent universels aux 3 archétypes ; exiger le socle web-saas d'un `landing`/`automation` = **faux-négatif** (`_shared/state-schema.md` §socle-par-archétype + portes par archétype de `routing.md`, ligne `12-build`). La passe **pose aussi le registre de solde** `tech/plan-ledger.md` (chaque `[SÉCU]` + tâche d'intégration du plan : *fait (fichier) / repoussé (raison tracée)*, tests du plan committés, ≥1 E2E cœur exécuté vert) — signé à l'étape 14 (`skills/14-qa/references/plan-solde-gate.md`, P0.1). Procédure, catalogue de jonctions et DoD dans `references/integration-pass.md`.
6. **Handoff cascade** — chaque feature dev-validée est **prête pour l'étape 13** (Tech Lead → CTO → Designer → CEO). Note l'état dans `status/`.

## Règles
- **1 feature = 1 worktree = 1 agent**, zones disjointes, communication **async par fichiers** (`status/`).
- **Jamais** plusieurs dev-agents sur la **même** zone en parallèle (collisions) — superpowers l'interdit, nous aussi.
- **Budget d'itération** (du plan / `_shared/validation-cascade.md`) sur le dev-loop — pas de boucle infinie.
- **tsconfig STRICT du châssis obligatoire** : chaque `feature-dev` boucle `tsc` (`npm run typecheck`, `noUncheckedIndexedAccess` inclus) → **0 erreur** avant de rendre sa feature. Interdit de contourner (`@ts-ignore`, `any`, tsconfig relâché).
- **Le gate de compilation ≠ `tsc` seul** : la porte de l'org lance le **build réel de l'archétype** (`next build` + lint pour `web-saas`) — `tsc` vert ne prouve pas que le build passe.
- **Build vert ≠ SQL correct** : la passe d'intégration exige en plus un **smoke-test de chaque RPC/fonction Postgres exposée contre une vraie base** (`supabase db reset` → appels réalistes chemin heureux + erreur métier → **0 erreur SQL runtime**, type `42702` « column reference … is ambiguous », RLS, grant). Ni `tsc` ni `next build` n'exécutent le SQL — seul un **appel réel** attrape le bug résa `create_booking` (`references/integration-pass.md`, § porte « le SQL tourne »).
- **Appliquer la charte AVANT tout** (1er geste du walking skeleton) : mapper les tokens de `DESIGN.md` dans `app/globals.css` (`:root`/`.dark`) **+** `tailwind.config`, en **écrasant** les défauts du châssis. Un produit au look **générique** du châssis (noir/blanc) = charte **non appliquée** → bug bloquant. Vérif : `app/globals.css` diffère des valeurs par défaut du châssis.
- **Brander l'identité AVANT tout** (même 1er geste, avec la charte) : renseigner `lib/brand.ts` (`name`, `tagline`, `description`) depuis `research/positioning.md` (nom + angle/tagline) **+** `product/pricing.md` (proposition de valeur). C'est la **source de vérité unique** du branding : `app/layout.tsx` (`<title>` / metadata), l'écran d'auth et la sidebar la consomment ; l'étape 16 (SEO) hérite du suffixe de titre via `title.template`. Le défaut `"SaaS Factory Template"` est une **sentinelle** : le laisser = `<title>` générique sur le site live (le dernier vestige châssis, invisible dans le corps de page) → bug bloquant. Ne **jamais** re-coder le nom du produit en dur ailleurs. Vérif : `lib/brand.ts` ≠ défaut châssis **et** `grep "SaaS Factory Template"` sur le repo = 0.
- **Générer les visuels on-brand AVANT le fan-out** (même 1er geste, après la charte — il faut `DESIGN.md` appliqué) : quand le design a besoin d'imagerie (hero éditorial/atmosphérique, **OG image**, empty-states, spot art, favicon/wordmark), le walking skeleton **GÉNÈRE** ces visuels via `scripts/generate-visual.mjs` (**Nano Banana Pro `gemini-3-pro-image`**, clé `GEMINI_API_KEY` lue de `~/.saas-factory/.env`), avec des **prompts dérivés de `DESIGN.md`** (section « Visuels générés » + palette/typo/métier) → sortie `public/generated/`, **câblée dans les composants via `next/image`** et l'**OG dans `metadata`** — **jamais du stock générique ni un placeholder** (levier anti-convergence : chaque projet a SES visuels). ⚠️ **Jamais une image IA maquillée en screenshot produit** (la preuve « voici l'app » reste un vrai screenshot — interdit doctrine n°17) ni en fausse preuve sociale (n°18). **Repli honnête** : `GEMINI_API_KEY` absente ou `providers.visuals="none"` → **ne pas simuler** d'images, consigner « ajoute ta clé », livrer le reste (`_shared/safety-rails.md` §6). Vérif : `public/generated/` peuplé + référencé `next/image`, `openGraph.images` pointe une OG générée, 0 stock/placeholder (passe d'intégration).
- **Retirer la tuyauterie exemple du châssis AVANT tout** (même 1er geste, avec la charte et le branding) : le châssis livre un **shell d'exemple** — un **gabarit à cloner, PAS le produit** — que le walking skeleton **supprime** dès qu'il pose la vraie surface. Il vise trois artefacts : **(1)** l'**entité CRUD d'exemple `items`** (`app/(app)/items/*`, `components/items/*`, `lib/schemas/item.ts`, migration `supabase/migrations/0002_items.sql`) — *le patron générique `lib/crud/factory.ts` **reste** (on le **clone** pour les vraies entités du PRD) ; c'est l'**instance `items`** qui disparaît* ; **(2)** le **dashboard d'exemple** `app/(app)/dashboard/page.tsx` (écran par défaut du châssis) ; **(3)** la **sidebar aux liens hardcodés** `components/nav/app-sidebar.tsx` (le tableau `navItems` liste en dur « Éléments » → `/items`, « Facturation » → `/billing`). **Régénère la navigation à partir des features du PRD** — les **routes réelles** du produit (`product/*`), jamais les liens en dur du châssis ; une entrée de nav ne pointe **que** vers une route qui **existe**. Les **redirections post-login** pointent vers le **vrai espace produit**, **jamais** un `/dashboard` d'exemple : repère-les toutes par `grep -rn "/dashboard" lib app` — la constante **`AFTER_LOGIN` (`lib/auth/actions.ts`, la cible du verify OTP + de la connexion par mot de passe — la principale)**, `app/(auth)/login/page.tsx` (déjà-authentifié), `lib/auth/get-user.ts` (`requireRole`), le callback auth (`app/auth/callback/route.ts`), le wordmark de la sidebar, et les URLs de retour Stripe si billing. **Aucun `/dashboard` en dur ne doit survivre** vers l'écran d'exemple supprimé (sinon le verify OTP atterrit sur un 404 alors même que la nav passe les greps). Laisser survivre ce shell = un fondateur non-tech découvre dans SON produit une entité « **Items — entité exemple du bloc CRUD, à cloner** » et un lien mort « **Facturation** » (404) — finding « Poser » → **bug bloquant**. Vérif : `find app -path '*items*'` = 0, sidebar dérivée du PRD, `grep -riE "à cloner|entité exemple|bloc CRUD" app components` = 0 (re-vérifié par le garde-fou anti-scaffolding de la passe d'intégration).
- **Billing/Facturation = nav CONDITIONNELLE** *(règle universelle « pas de lien mort »)* : un lien vers la facturation/billing n'existe **QUE si `billing=stripe`** est configuré (`providers.billing`, `~/.saas-factory/config.json` ; garde runtime `isBillingEnabled()`) — sinon **ni page ni lien**. Idem **toute feature optionnelle non livrée** (billing, réglages avancés, admin…) : **pas de lien** tant que la route n'existe pas. Un href de nav pointe toujours vers un `page.tsx` réel — **jamais** vers une 404. *(Vérifié à la passe d'intégration, rejoué au faux-client 14, re-prouvé en live 17.)*
- **La landing est un livrable à part entière** *(type public)*, pas un hero nu : vraie page marketing (hero + *comment ça marche* + features + *pricing* + CTA + footer), dérivée de `research/positioning.md` + `product/pricing.md` + PRD (+ clusters SEO de l'étape 16). Traite-la comme une feature avec critères d'acceptation. La référence normative est `_shared/landing-playbook.md` (11 sections canoniques, règles de copy, légal adapté à la juridiction `jurisdiction`/`locale`) : sa checklist binaire fait partie des critères d'acceptation de la landing. **Calibré par type** : en **interne/perso**, **pas de landing marketing** — la racine est le **login** et le gate devient « pas de template châssis visible » + `noindex` (**route selon `skills/saas-factory/references/routing.md`**, ligne `12-build` ; ne recopie pas la matrice).
- **Pricing = features livrées** *(type public, P0.7)* : la page pricing ne vend **que** des capacités **effectivement livrées**. Diff chaque tier/feature contre la **liste des Must implémentés** du PRD ; une feature vendue mais absente du code = badge « **bientôt** » **ou** retirée de la carte ; une feature **structurellement impossible** (ex. « multi-praticiens » quand le schéma l'interdit) = **retirée**. **Zéro vaporware public** — la promesse commerciale est vraie par construction. *(Vérif à la passe d'intégration ; re-vérifiée au faux-client, 14, et par le persona du client-review, 15.)*
- **Boucles fermées — aucune action de valeur muette** (`_shared/boucles-fermees.md`) : pour chaque action de valeur du workflow cœur (créer/modifier/annuler l'entité métier), la boucle dérivée en 07 est **câblée et testable** — l'email/notif part **dans la même transaction logique** que l'action (pattern : job `notification_jobs` + envoi immédiat best-effort ; **jamais « TODO plus tard »**). L'acteur reçoit sa trace durable, la contrepartie est notifiée. **Rappel planifié = cron QUOTIDIEN par défaut** (`"0 8 * * *"`, compatible Vercel Hobby) dont le worker balaie les échéances des ~24-48 h à venir, idempotent via `notification_jobs` ; une cadence sub-quotidienne (`*/10`, « H-2 ») **exige Vercel Pro** et **fait échouer le déploiement sur Hobby** — jamais le défaut (`_shared/boucles-fermees.md`). La passe d'intégration vérifie **zéro action muette** ; le faux-client (14) rejoue les deux rôles.
- **L'onboarding est une feature Must implicite de tout `web-saas`** *(socle web-saas — conditionné par archétype)* — jamais livrer un web-saas qui atterrit sur du vide. Le parcours de premier usage fait partie du périmètre du build : **signup OTP → mot de passe** (code OTP de vérif e-mail puis mot de passe posé ; jamais mot de passe **sans vérif OTP préalable**, plus de magic link) → **onboarding qui crée l'entité cœur** du produit (ex. profil salon : nom, adresse, horaires, prestations) → **dashboard d'arrivée non-vide** (il reflète l'entité créée) ; chaque liste encore vide porte un **empty state avec CTA**. Si le plan (étape 10) ne l'a pas listé, le Tech Lead l'ajoute comme lane Must. **Ne s'applique PAS** hors `web-saas` : un `landing` n'a **pas** d'onboarding/dashboard (socle **LP1-LP4** : sections + légal + waitlist/CTA + métadonnées), un `automation` est **headless** (socle **AU1-AU5** : config + runs/logs + healthcheck + boucle fermée + idempotence) — `_shared/state-schema.md` §socle-par-archétype, portes par archétype dans `routing.md`.
- **Pas de secret** en dur ni loggé.

## Sortie & état
Code buildé + tests verts par feature + `status/*` à jour (template `assets/templates/feature-status.md`). Mets à jour `.saas-factory/state.md`. Résume (features construites + intégration). Annonce l'**étape 13** (cascade Tech Lead → CTO → Designer → CEO).
