L'étape 5 de la procédure. Le parallélisme a un coût caché : les **bugs de jonction** (`_shared/lessons.md`, anti-pattern « parallélisme non maîtrisé »). Un build qui parallélise **sans** passe d'intégration ment sur son état final. Piloté par le Tech Lead.

## Flux merge → intégration
```
lanes DEV-DONE (status/*.md)         Chaque lane part de main (skeleton),
   A ─┐                              zones disjointes.
   B ─┤   merge SÉQUENTIEL sur main
   C ─┘   dans l'ordre A → B → C
              │
              ▼
     conflit git ?
       ├─ oui → résous (jamais d'écrasement aveugle) → re-run tests
       └─ non → suite complète verte ?
                    ├─ non → bug de jonction → tâche d'intégration
                    └─ oui → lane suivante
              │
              ▼
     PASSE D'INTÉGRATION (dédiée) : parcours A→Z qui traverse les features mergées
              │
              ▼
     DEV-DONE consolidé → prêt pour l'étape 13 (cascade)
```

## Procédure
1. **Pré-check** : lis tous les `status/*.md`. Ne merge que les lanes `DEV-DONE` (une lane `BLOCKED` attend, ne bloque pas les autres).
2. **Merge séquentiel** (jamais deux merges concurrents) : intègre une lane, fais tourner la **suite complète**, puis la suivante. L'ordre suit les dépendances (le contrat/skeleton d'abord).
3. **Conflits git** : résous à la main (jamais « accepter les leurs/nôtres » à l'aveugle sur du code métier). Re-teste après résolution.
4. **Bugs de jonction** : deux features vertes séparément peuvent casser ensemble (types divergents, contrat d'API mal aligné, effets de bord partagés). Traite via une **tâche d'intégration** dédiée.
5. **Passe d'intégration** : lance un parcours **A→Z** qui traverse plusieurs features mergées (pas les unitaires de chaque lane) — le chemin cœur + les greffes.
6. **Consolide** : mets `status/*` à jour (DEV-DONE consolidé) ; les bugs résolus, les réserves restantes en `CONCERNS` (l'étape 13/14 tranchera).

## Catalogue de bugs de jonction (à chercher activement)
| Type | Exemple niche-agnostique | Où ça casse |
|---|---|---|
| **Contrat divergent** | Lane A renvoie `{id}` , lane B attend `{itemId}` | Frontière API/types |
| **Schéma partagé** | Deux migrations touchent la même table dans un ordre incompatible | BDD |
| **Effet de bord global** | Deux features écrivent le même cache/état global | Runtime |
| **Auth / middleware** | Une lane suppose un middleware qu'une autre a modifié | Pipeline de requête |
| **Dépendance de version** | Deux lanes bumpent la même lib vers des versions différentes | Build/lockfile |
| **Route en collision** | Deux features déclarent la même route/chemin | Routeur |
| **Ordre d'init** | B suppose une ressource initialisée par A, pas garantie au boot | Démarrage |

## Definition-of-Done de la passe d'intégration
- [ ] Toutes les lanes `DEV-DONE` mergées sur `main` (les `BLOCKED` documentées, pas oubliées).
- [ ] **Suite complète verte** après le dernier merge (0 régression).
- [ ] Au moins **un parcours A→Z** vert qui traverse les features greffées (pas seulement les unitaires par lane).
- [ ] **Smoke-test SQL vert** : chaque RPC/fonction exposée appelée ≥1 fois (chemin heureux + erreur métier) contre une vraie base (`supabase db reset` local ou staging) → **0 erreur SQL** (42702 & co.) ; `tsc`/`next build` verts **ne suffisent pas** (§ smoke-test des fonctions/RPC).
- [ ] Chaque conflit git résolu **et re-testé** (pas d'écrasement aveugle).
- [ ] Catalogue de jonction parcouru ; bugs corrigés ou logués en `CONCERNS`.
- [ ] **Porte MACHINE — `npm run verify:machine` VERT** (slop + sentinelle + secrets + SQL 42702-class + i18n) **+ `tsc`/`next build` verts + `npm audit` high = 0** — c'est le **PLANCHER exécutable**, vérifié **AVANT tout jugement d'agent** ; un garde-fou jugé « passé » par un agent alors que `verify:machine` est **rouge** = **non-passé** (la machine est le plancher, l'agent le plafond — jamais l'inverse ; `_shared/lessons.md` §18, `_shared/blocks/web-saas/scripts/`). Les garde-fous de sortie ci-dessous **n'ouvrent que sur du vert machine**.
- [ ] **Garde-fous de sortie produit** verts (charte + anti-slop / **visuels générés on-brand** *(si imagerie)* / landing playbook *(public)* / **pricing = livré** *(public)* / **zéro tuyauterie exemple** *(anti-scaffolding)* / métadonnées + favicon + OG générée / premier usage complet *(socle web-saas)* / **boucles fermées** / empty states *(socle web-saas)* / **légal adapté à la juridiction** *(public)* / 404 + loading — cf. section suivante). **Calibrés par type ET conditionnés par archétype** : les items du **socle web-saas** (premier usage/onboarding/dashboard/empty states) ne s'exigent que si `archetype=web-saas` — un **`landing`** vérifie le socle **LP1-LP4** (sections landing-playbook + légal + waitlist/CTA + métadonnées/OG), un **`automation`** le socle **AU1-AU5** (config/secrets + runs/logs + healthcheck + boucle fermée + idempotence). `_shared/state-schema.md` §socle-par-archétype + `skills/saas-factory/references/routing.md` (portes par archétype, ligne `12-build` ; interne/perso : pas de landing, racine=login, noindex).
- [ ] **Zéro tuyauterie exemple** (anti-scaffolding, bloquant, universel) : entité `items` non atteignable (`find app -path '*items*'` = 0, `/items` → 404), aucun texte de dev rendu (les **phrases-sentinelles** de l'exemple : « à cloner », « entité exemple », « bloc CRUD », « lorem »), aucun lien de nav mort (chaque href → route existante ; `/billing` **seulement** si `billing=stripe`), **aucune redirection serveur vers un écran supprimé** (`AFTER_LOGIN`/`redirect()`/URLs Stripe → route existante ; `grep -rn "/dashboard" lib app` ne pointe plus vers l'écran d'exemple retiré). Rejoué par le faux-client (14), re-prouvé sur la prod (17, « route 200 ≠ feature »).
- [ ] **Boucles fermées** (`_shared/boucles-fermees.md`) : chaque action de valeur enfile+envoie sa notification (acteur + contrepartie), **zéro action muette** ; testable par le faux-client (étape 14).
- [ ] **Registre `tech/plan-ledger.md` posé** (P0.1) : chaque `[SÉCU]` + tâche d'intégration du plan tracé *fait (fichier) / repoussé (raison + décision humaine)* ; tests du plan **committés** ; ≥1 **E2E cœur exécuté vert** (trace) — prêt à être signé par l'étape 14 (`skills/14-qa/references/plan-solde-gate.md`).
- [ ] `status/*` consolidé ; secrets absents ; état prêt pour l'étape 13.

## Garde-fous de sortie produit (non-régression du « générique châssis » et du « squelette »)
Le parallélisme livre des features vertes mais peut laisser sortir un produit au **look châssis**, au **contenu squelette** ou au **premier usage vide**. Ces checks sont **bloquants** — un seul échec = le produit n'est pas prêt pour l'étape 13, même si la suite est verte. Ils vérifient que les **gestes de fondation** du walking skeleton ont tenu après tous les merges, et que le produit sort **complet et pro**, pas en démo creuse. Règle de fond : **l'onboarding est une feature Must implicite de tout `web-saas` — jamais livrer un web-saas qui atterrit sur du vide.** Mais le socle est **conditionné par archétype** (axe orthogonal au `type` ; `_shared/state-schema.md` §socle-par-archétype) : premier usage/onboarding/dashboard/empty states sont le **socle web-saas** ; un **`landing`** (pas d'auth/dashboard) se juge sur le **socle LP1-LP4**, un **`automation`** (headless) sur le **socle AU1-AU5** — voir l'encart « Calibrage par archétype » sous la table. Exiger le socle web-saas d'un landing/automation = **faux-négatif** (le bug que ce conditionnement corrige).

## Porte anti-scaffolding — zéro tuyauterie exemple du châssis dans le produit livré
Le châssis livre un **shell d'exemple** (entité CRUD `items`, dashboard d'exemple `app/(app)/dashboard`, sidebar aux liens hardcodés `components/nav/app-sidebar.tsx` : « Éléments » → `/items`, « Facturation » → `/billing`) : un **gabarit à cloner**, retiré au walking skeleton (`walking-skeleton.md` §0 bis). Le parallélisme peut le laisser **ressusciter** — le build crée le vrai espace produit **à côté** mais **oublie de supprimer** le shell d'exemple. Résultat vécu (finding « Poser ») : un fondateur non-tech découvre dans SON produit une entité « **Items — entité exemple du bloc CRUD, à cloner pour vos propres entités** » et un lien de nav « **Facturation** » vers `/billing` **inexistant (404)**. Inacceptable pour un non-tech. **Porte bloquante**, en quatre checks grep-ables + rejouables :

1. **Entité exemple injoignable** — aucune route/écran `items` : `find app -path '*items*'` → **0** (route supprimée) **et** une requête `/items` renvoie **404**. Idem tout artefact d'exemple (`components/items/*`, `lib/schemas/item.ts`, migration `0002_items.sql`) retiré ou remplacé par une **vraie** entité du PRD. *(Le patron générique `lib/crud/factory.ts` reste — on le clone ; c'est l'instance `items` qui doit disparaître.)*
2. **Aucun texte de dev rendu** — `grep -riE "à cloner|entité exemple|bloc CRUD|lorem ipsum" app components` sur les surfaces produit → **0** (le châssis rendait « *Entité exemple du bloc CRUD — à cloner pour vos propres entités* » : c'est exactement ce que l'utilisateur ne doit **jamais** lire). *(On grep les **phrases-sentinelles** de l'exemple, PAS le mot nu `placeholder`/`TODO` : ceux-ci matchent la classe Tailwind `placeholder:`, l'attribut `placeholder=` de tout `<input>` et les commentaires `// TODO` — tous légitimes → le gate ne pourrait jamais valider `= 0`. Un « TODO » ou une lorem ipsum **visible à l'écran** est du ressort du faux-client 14, pas d'un grep source.)*
3. **Aucun lien de nav mort** — chaque `href` interne de la nav (`components/nav/app-sidebar.tsx`, `components/nav/top-nav.tsx`) pointe vers une route **qui existe** (un `page.tsx`). La nav est **dérivée des features du PRD**, pas des liens en dur du châssis. Le lien **Facturation** (`/billing`) n'existe **que si `billing=stripe`** est configuré (`providers.billing`, `~/.saas-factory/config.json` ; garde runtime `isBillingEnabled()`) — sinon **ni page ni lien**. Règle générale : **toute feature optionnelle non livrée = pas de lien** (jamais un href vers une 404).
4. **Aucune redirection serveur vers un écran d'exemple supprimé** *(le check 3 ne voit QUE `components/nav/*` — pas les `redirect()` serveur)* — après retrait de l'écran d'exemple `app/(app)/dashboard`, plus **aucune** cible `/dashboard` en dur ne doit pointer vers cette route morte : `grep -rn "/dashboard" lib app` → chaque occurrence est **repointée vers le vrai espace produit**. Sources concernées : **`AFTER_LOGIN` (`lib/auth/actions.ts`) — la cible du verify OTP + de la connexion par mot de passe, donc le parcours d'auth par défaut** —, `app/(auth)/login/page.tsx` (déjà-authentifié), `lib/auth/get-user.ts` (`requireRole`), le callback (`app/auth/callback/route.ts`), le wordmark de la sidebar, et les `success_url`/`cancel_url`/`return_url` Stripe si billing. **Piège vécu** : un build supprime le dashboard d'exemple et régénère la nav (checks 1-3 verts) mais laisse `AFTER_LOGIN = "/dashboard"` → **le premier login réel atterrit sur un 404**, invisible aux trois premiers checks. Idéalement, la cible post-login est **une seule constante** que toutes ces surfaces importent (pas 6 littéraux `/dashboard` dupliqués).

C'est le pendant **amont** du **principe cardinal de la recette live — « route 200 ≠ feature qui marche »** (`skills/17-deploy/references/live-qa.md`) : ici on exige le **symétrique** — *aucun lien ne mène à une non-feature*, et *aucune non-feature (le shell d'exemple) n'est atteignable*. Le **faux-client** (14) le rejoue en clic réel : il parcourt toute la nav en persona et **échoue** s'il atterrit sur un écran d'exemple, un texte de dev, ou un 404 (`skills/14-qa/references/fake-client-protocol.md`). La preuve monte ensuite en live (17), sur la prod réelle.

## Porte « le SQL tourne » — smoke-test des fonctions/RPC contre une vraie base
**`tsc` vert + `next build` vert NE PROUVENT PAS que le SQL marche.** Une fonction Postgres (`plpgsql`, RPC exposée) n'est ni compilée ni exécutée au build — elle est résolue **à l'exécution**. Une erreur runtime — collision colonne/variable **42702**, `search_path` manquant, type de retour, RLS qui refuse l'appel — franchit **toute la CI verte** et explose au **premier vrai appel en prod**. C'est le **chaînon manquant** entre « build vert » et « ça marche » : le bug réel `create_booking` (42702 « column reference `hold_expires_at` is ambiguous », `skills/09-architecture/references/data-model.md §Robustesse des fonctions PL/pgSQL`) a passé `tsc` **et** `next build` — seul un **appel réel** l'aurait attrapé.

**Exigence de porte (bloquante).** Avant de consolider `DEV-DONE`, **chaque RPC/fonction exposée est appelée au moins une fois** avec des **données réalistes**, contre une **vraie base** — Supabase local (`supabase db reset` puis appels) **ou** la base de staging — et retourne **0 erreur SQL**.
- **Cible** : toutes les fonctions/RPC que le front appelle (`free_slots`, `create_booking`, `confirm_booking`, purge, notifs…), pas seulement celles du parcours cœur.
- **Réaliste** : au moins un appel du **chemin heureux** (slot libre → réservé) **et** un du **chemin d'erreur métier** (slot déjà pris → l'exception *attendue*, pas un 42702).
- **`0 erreur SQL`** = aucune 42702/42703/… ; une **exception métier attendue** (`raise exception 'slot indisponible'`) est un **succès** du test d'erreur, pas un échec.
- **Preuve par exécution** (même exigence que la porte plan-soldé ci-dessous, P0.1) : la **trace** des appels (sortie `psql` / test d'intégration DB committé) est jointe — jamais « le code a l'air bon ».

> `supabase db reset` recrée la base **locale** depuis les migrations, puis on appelle les RPC : on vérifie d'un coup que **la migration s'applique** ET que **la fonction tourne**. (`supabase db reset --linked` vise la base **DISTANTE** — réservé à un sandbox, double-confirmation `safety-rails §5` ; **jamais** sur une prod.)

## Porte « plan soldé » — le producteur du registre (P0.1)
La passe d'intégration est le **producteur** du registre de solde ; l'étape **14** en est le vérificateur qui l'**exécute et le signe**. **Critères canoniques : `skills/14-qa/references/plan-solde-gate.md`** (source unique — ne pas les redéfinir ici). Ce que la passe d'intégration **fait** :

1. **Relis `tech/execution-plan.md`** : recense **chaque** item `[SÉCU]` et **chaque** tâche d'intégration.
2. **Pose `tech/plan-ledger.md`** : une ligne par item — `fait (fichier:ligne / test)` **ou** `repoussé (raison tracée + décision humaine loguée `state.md` + `CONCERNS`)`. **Aucun `[SÉCU]` sans ligne** (c'est là qu'ils s'évaporent). Un `[SÉCU]` `fait` **exige** une référence fichier, pas une relecture.
3. **Committe les tests du plan** : les rouges du TDD (étape 10/12) et recettes que le plan nomme deviennent des **fichiers versionnés** (`git ls-files` les trouve) — pas « décrits ».
4. **Exécute ≥1 E2E du parcours cœur** et joins la trace verte : la preuve est l'**exécution**, jamais la lecture. *(L'E2E `upgrade/billing` n'est requis que si public + billing — `skills/saas-factory/references/routing.md`.)*

> Un `[SÉCU]` repoussé ne se **`WAIVE` jamais** sur épuisement de budget sans décision humaine explicite (`_shared/validation-cascade.md`). Le registre part à l'étape 14, qui **ferme la porte** si un `[SÉCU]` manque, si les tests ne sont pas committés, ou si aucun E2E cœur n'a tourné.

## Matrice de décision
| Condition | Action |
|---|---|
| Suite verte après un merge | Lane suivante |
| Rouge après un merge (feature seule était verte) | Bug de jonction → tâche d'intégration ciblée |
| Conflit git sur du code métier | Résous à la main + re-teste ; jamais « ours/theirs » aveugle |
| Deux lanes bumpent une lib différemment | Aligne sur une version, re-teste les deux |
| Bug de jonction non résolu dans le budget | Loge en `CONCERNS` + remonte ; l'humain tranchera (étape 15) |
| Une lane est `BLOCKED` | Merge les autres ; ne bloque pas tout ; traite le blocage à part |

## Forcing-question — « l'intégration est-elle prouvée ? »
- **Ask exact** : *« Existe-t-il un parcours A→Z exécuté qui traverse PLUSIEURS features mergées, vert, sur main à jour ? »*
- **Push-until** : validé quand un test de parcours transverse passe **après** le dernier merge — pas la somme des tests par lane.
- **Red-flags** :
  - « chaque feature était verte » (séparément ≠ ensemble),
  - « ça compile » (compiler ≠ le chemin cœur tourne),
  - conflit résolu sans re-lancer les tests,
  - pas de test transverse, seulement des unitaires par lane.
- **MOU vs FORT** :
  - MOU : *« Tout est mergé, ça build. »*
  - FORT : *« A,B,C mergées ; suite 84/84 verte ; e2e ‘créer → commenter → notifier' vert sur main. 1 bug de contrat corrigé (B attendait itemId). »*

## Modes d'échec
| Mode d'échec | Symptôme | Parade |
|---|---|---|
| **Pas de passe d'intégration** | On déclare fini sur la somme des lanes | Toujours une passe A→Z dédiée après merge |
| **Merge concurrent** | Deux merges en //, historique cassé | Merge **séquentiel**, une lane à la fois |
| **Résolution aveugle** | `ours/theirs` sur du métier → perte de code | Résolution manuelle + re-test |
| **Faux « fini »** | Vert par lane, rouge ensemble non détecté | Test transverse obligatoire dans la DoD |
| **Blocage propagé** | Une lane BLOCKED gèle le merge de tout | Merge les DEV-DONE, isole le blocage |
| **Sortie générique** | Le `<title>` live affiche encore « SaaS Factory Template » (ou look/landing châssis) | Garde-fous de sortie produit : `lib/brand.ts` renseigné au walking skeleton + gate `grep "SaaS Factory Template"` = 0 |
| **Produit squelette** *(socle web-saas)* | Features vertes mais le premier usage atterrit sur du vide (pas d'onboarding, dashboard vide, listes nues, légal absent) | Garde-fous « premier usage / empty states / légal / 404 » **du socle web-saas** : le parcours OTP → onboarding → entité cœur → dashboard non-vide est **joué** avant l'étape 13, pas supposé. **Conditionné archétype** : landing → socle **LP1-LP4**, automation → socle **AU1-AU5** (`_shared/state-schema.md` §socle-par-archétype) |
| **Faux-négatif d'archétype** | Un `landing` (pas de dashboard) ou un `automation` (headless) est recalé pour « onboarding/dashboard manquant » alors qu'il est conforme à SON socle | On vérifie le **socle de l'archétype**, pas le socle web-saas d'office : landing → **LP1-LP4** (sections + légal + waitlist + OG), automation → **AU1-AU5** (config + runs/logs + healthcheck + boucle fermée + idempotence) — `_shared/state-schema.md` §socle-par-archétype + portes par archétype de `routing.md` |
| **Action de valeur muette** | Une action cœur (résa, commande, annulation) confirme à l'écran mais **aucune notification** ne part (job jamais consommé, « TODO plus tard ») | Garde-fou **boucles fermées** : chaque action enfile+envoie sa notif (acteur + contrepartie) dans la même transaction logique ; vérifié des deux côtés au faux-client (`_shared/boucles-fermees.md`) |
| **Confirmation différée au cron** | La notif **existe** (fonction d'envoi ciblé codée) mais n'est **appelée nulle part** au site de l'action → le job reste `pending` et n'est livré qu'au prochain cron quotidien (jusqu'à ~24 h de retard) — finding « Poser » : `dispatchAppointmentJobs(id)` définie, jamais invoquée dans le handler de réservation | Garde-fou **boucles fermées** — check du **call-site** : `grep` que l'envoi immédiat (`dispatchXJobs(id)`) est **invoqué** dans chaque handler create/confirm/cancel (pas seulement défini) ; le cron ne porte QUE J-1 + retry. Le faux-client (14) mesure l'**immédiateté** (`status=sent` en quelques s, avant tout cron) |
| **Funnel muet / boucle de mesure ouverte** | Le catalogue d'events est **défini** mais aucun `capture()` n'est **appelé** au call-site → 18-metrics lit un funnel vide au 1er run, pilotage aveugle (bug « AgencyDesk » : `events.ts` générique, `capture()` seulement sur `$pageview`) | Garde-fou **boucle de mesure** — check du **call-site** (miroir des boucles fermées) : `grep "capture("` hors `lib/analytics/` → moment magique **invoqué** ; `identify()`/`resetIdentity()` câblés ; conditionné par type/archétype (funnel public/interne/perso, `waitlist_joined` landing, métriques de run automation). Le faux-client (14) vérifie l'**émission réelle** en staging |
| **Vaporware pricing** | La landing vend un tier/feature que le code ne livre pas (ou que le schéma interdit) | Garde-fou **pricing = livré** *(public)* : diff pricing ↔ Must implémentés ; feature absente = « bientôt » ou retirée (P0.7) |
| **Shell d'exemple ressuscité** | Le produit livré expose encore l'entité `items` (« à cloner »), un dashboard d'exemple, ou un lien de nav mort (« Facturation » → `/billing` 404) — le vrai espace produit a été créé **à côté** sans retirer le shell châssis (finding « Poser ») | **Porte anti-scaffolding** : entité exemple injoignable (`find app -path '*items*'`=0, `/items`→404) + 0 texte de dev rendu + 0 lien mort (nav dérivée du PRD, `/billing` seulement si `billing=stripe`) ; rejouée au faux-client (14), re-prouvée en live (17) |
| **`[SÉCU]` évaporé / faux « fini »** | Un item `[SÉCU]` du plan absent du code sans trace ; « c'est fait » sans test ni E2E | **Registre `tech/plan-ledger.md`** : une ligne par `[SÉCU]` (fait fichier / repoussé tracé), tests committés, E2E cœur exécuté — la porte 14 ferme sinon (`skills/14-qa/references/plan-solde-gate.md`) |
| **Build vert, SQL cassé** | `tsc` + `next build` verts, mais une RPC (`create_booking`) lève **42702** au premier appel réel (bug résa « Poser ») | **Smoke-test des fonctions/RPC** : chaque RPC appelée ≥1 fois contre une vraie base → 0 erreur SQL ; le build ne voit pas le SQL (§ smoke-test des fonctions/RPC) |
