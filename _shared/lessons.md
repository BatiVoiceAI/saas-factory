# Lessons — les règles d'or de la méthode (lues par chaque sous-skill au démarrage)

Signature méthodologique, extraite de projets réels (bativoice en prod, speechtoflow shippé macOS+Windows, evalprof, Spendly) + du premier run SaaS Factory déployé de bout en bout (2026-07). À appliquer partout.

## Règles d'or
1. **Git dès J0.** Jamais de versionnement par copie de dossier.
2. **Valider le marché avant de builder.** La porte Go/No-Go est bloquante. Ne pas « construire en stealth sans un seul retour cible ».
3. **Time-box la spec.** Un walking skeleton tôt > une doc parfaite. Sur-planification = gel.
4. **1 feature = 1 worktree = 1 agent**, zones de code disjointes, communication asynchrone par fichiers.
5. **Le Verifier resynchronise la doc**, pas seulement les tests (éviter que la doc décroche du code quand les agents itèrent vite).
6. **Vérification adversariale** sur toute sortie d'agent : sources en tiers (1/2/3), score de confiance, verdict honnête. Jamais de fausse certitude.
7. **Éval-driven pour l'IA** : jeux de tests « gold », télémétrie par version de prompt.
8. **Un seul standard** de specs/plans datés. Pas de conventions ad hoc qui divergent.
9. **Kill explicite.** Un critère écrit déclenche l'archivage d'une piste morte + un post-mortem de 5 lignes.
10. **La venture = 5 briques** (Produit, Infra, Marketing, Stratégie, Intel), pas un simple repo de code.
11. **(2026-07) La complétude est une exigence de spec, pas un polish.** Un SaaS aux features correctes mais sans onboarding, sans profil ni empty states paraît creux à l'usage. Le socle « produit complet » (onboarding qui crée l'entité cœur, profil/settings, empty states, emails brandés, légal adapté à la juridiction du produit — jamais « FR » en dur, 404, seed data) entre en Must au PRD (étape 7) — il ne se rattrape pas en fin de build.
12. **(2026-07) Le design par défaut du châssis = AI slop perçu.** Sans contraintes imposées AVANT génération, le rendu régresse vers la médiane (gradient indigo, Inter partout, hero centré). `design-doctrine.md` + ses recettes sont obligatoires sur chaque page, pas optionnelles.
13. **(2026-07, révisé) Standard d'auth = OTP (vérif e-mail à l'inscription) → création d'un mot de passe → connexion e-mail + mot de passe. PLUS DE magic link (décision produit Felix).** L'inscription envoie un **code OTP** qui **vérifie l'e-mail** (`signInWithOtp` pour émettre le code → `verifyOtp` qui ouvre la session), puis l'utilisateur **pose un mot de passe** (`updateUser`) ; le **retour** se fait en **e-mail + mot de passe** (`signInWithPassword`) ; « mot de passe oublié » repasse par un **OTP** → nouveau mot de passe. **Aucun magic link** dans l'UI ni dans l'e-mail — le mail d'OTP ne porte QUE le code (`{{ .Token }}`, **sans** `{{ .ConfirmationURL }}`). Avantage conservé : l'**e-mail est vérifié par construction** (l'OTP prouve la possession de la boîte avant même que le mot de passe soit posé). ⚠️ Ceci **INVERSE** la version passwordless d'origine de cette leçon (« email + mot de passe jamais le défaut ») : c'est désormais le standard, assumé.
14. **(2026-07) Build vert ≠ SQL correct.** `tsc` et `next build` ne compilent ni n'exécutent le SQL : une fonction Postgres peut passer toute la CI verte puis exploser au **premier appel réel** en prod (bug réel « Poser » : erreur runtime **42702** « column reference `hold_expires_at` is ambiguous » dans `create_booking`). **Toute fonction/RPC doit être exécutée au moins une fois** contre une vraie base (Supabase local `supabase db reset` ou staging) **avant livraison** — c'est le smoke-test SQL de la passe d'intégration (`skills/12-build/references/integration-pass.md`), le chaînon manquant entre « build vert » et « ça marche ».
15. **(2026-07) Les colonnes OUT de `RETURNS TABLE` collisionnent avec les colonnes de table.** Une sortie `RETURNS TABLE(… hold_expires_at …)` et la colonne `appointments.hold_expires_at` portent le même nom → toute référence **nue** est ambiguë à l'exécution (42702), invisible au build. Parade normative sur **toute** fonction `plpgsql` (surtout `SECURITY DEFINER`/`RETURNS TABLE`) : `#variable_conflict use_column` en tête de corps **ou** variables préfixées `v_`/`p_`/`c_` + **toute colonne qualifiée** (`table.colonne`) — jamais un nom nu collisionnable (`skills/09-architecture/references/data-model.md §Robustesse des fonctions PL/pgSQL`).
16. **(2026-07) Enfiler un job de notification ≠ l'envoyer — le cron ne porte jamais la 1ʳᵉ livraison d'une confirmation.** Une boucle fermée a **deux temps indissociables** : (a) enfiler le job en `notification_jobs` (durabilité/idempotence/retry) ; (b) **appeler l'envoi immédiat de ce dossier AU SITE DE L'ACTION** (handler create/confirm/cancel, best-effort). Écrire la fonction d'envoi ciblé et **oublier de l'appeler** = boucle muette : le job dort jusqu'au prochain cron quotidien, le client attend jusqu'à ~24 h (bug réel « Poser » : `dispatchAppointmentJobs(id)` **définie mais jamais invoquée** dans le handler de réservation → confirmation cliente + notif gérant restées `pending`). Le **cron** ne sert QU'À deux choses : rappels planifiés (J-1) + retry des `pending` en échec. Vérif normative : `grep` du **call-site** (envoi invoqué, pas seulement défini) à la passe d'intégration (12) ; la QA (14) et la recette live (17) mesurent l'**immédiateté** (`status=sent` en quelques secondes, avant tout cron) — `_shared/boucles-fermees.md`.
17. **(2026-07) L'auteur du commit git doit être membre de la team de l'hébergeur (Vercel), sinon le déploiement est BLOCKED — silencieusement.** Vercel refuse (`readyState = BLOCKED`, **pas** ERROR — build vert : `tsc`/`next build` passent, le déploiement est juste **refusé**) tout déploiement dont l'**auteur du commit** (email) n'est pas membre de la team ; message exact « Git author <email> must have access to the team <team> on Vercel to create deployments ». 🚨 **INVARIANT — `git_author.email` DOIT être un membre de la team de l'hébergeur du compte de déploiement** : le domaine/l'adresse importe peu, seule l'appartenance à la team est dure. Bug réel « Poser » : commit authored par l'email git par défaut de la machine (`felix@bativoiceai.com`) → **BLOCKED** ; même `factory@speechflow.fr` restait BLOCKED ; **seul** l'email du compte membre de la team (`no_reply@speechflow.fr`) déploie (READY). Correctif à chaud : `git commit --amend --reset-author` avec l'email membre + re-deploy. **Parade normative** : miroir exact d'`email_from` — l'identité git-author est une **question de l'onboarding** (infra-setup, `config.git_author {name, email}`), **appliquée au repo projet au scaffold (étape 11 : `git config user.email` / `user.name`)** et **vérifiée en pré-vol (étape 17)** avant promotion — jamais un défaut deviné en cours de run (`skills/infra-setup/references/config-schema.md §git_author`).

18. **(2026-07) La vérif MACHINE est le plancher, le jugement d'agent est le plafond — jamais l'inverse.** Une porte cochée par un LLM sans backstop grep/exécutable est **fragile** : l'agent peut se convaincre qu'une checklist « passe » (anti-slop, boucles fermées, complétude « pro vs démo ») alors qu'elle ne passe pas. **Toute qualité automatisable** — slop, copy sentinelle, secrets en dur, SQL 42702-class, i18n (légal/emails/copy pas « FR » en dur), `tsc`/`next build`, `npm audit` high — **DOIT avoir un check machine qui échoue en ROUGE, exécuté AVANT le jugement d'agent**. L'agent ne juge **que ce que la machine ne sait pas juger** (goût, cohérence métier, vécu du parcours), **jamais l'inverse** : un garde-fou jugé « passé » par un agent alors que la machine est rouge = **non-passé**. Point d'entrée unique : `npm run verify:machine` (`_shared/blocks/web-saas/scripts/`), câblé en plancher aux portes 12 (`integration-pass.md` DoD), 14 (`fake-client-protocol.md`) et 17 (`preflight-checklist.md`) — défini **une seule fois**, renvoyé partout.

19. **(2026-07) Rien n'est « livré » tant que la recette live AUTHENTIFIÉE n'a pas exécuté CHAQUE action de valeur RLS-protégée de CHAQUE rôle sur la PROD — avec preuve.** Une route 200, une landing OK et un login OTP qui **s'affiche** ne prouvent **rien** du parcours **CONNECTÉ** : les bugs **RLS / scoping / junction** ne surgissent QUE pour un utilisateur **authentifié** qui exécute les actions protégées (bugs vécus AgencyDesk, 2026-07 : « new row violates RLS policy for table `orgs` » PUIS `projects` à la **1ʳᵉ action connectée** + parcours d'**invitation client injoignable** — tous passés inaperçus car la recette ne **complétait pas le parcours CONNECTÉ**). Une recette **« à l'écran »** ou sur **données seedées** les **masque**. Parade normative — **étape de 1er rang `17b` (Phase 5), obligatoire & bloquante APRÈS le canary** : sur la PROD réelle, **franchir vraiment l'auth** (login OTP **complété** via **boîte sandbox de test** OU session **forgée par l'Admin API** : `createUser` `email_confirm` + `signIn` — technique **réutilisable et documentée**), puis pour **chaque rôle** (multi-rôles : ex. agence + client, org A + org B) exécuter **chaque action de valeur RLS-protégée** avec **preuve** : réponse **2xx** + **ligne écrite sous le bon tenant** + **notification `sent`** (immédiateté, cf. règle 16) + **refus cross-tenant prouvé** (un rôle ne voit/écrit pas les données d'un autre). Boucle **fix → redeploy → re-test** jusqu'au vert ; tant que ce n'est pas vert, on **ne déclare PAS « livré »**. Applicabilité par archétype × type × tenancy (dont `perso` = action authentifiée sans cross-tenant, `landing` = sans objet, `automation` = boucle fermée) : `skills/phase-5-launch/references/routing.md` §7. Câblée : `skills/phase-5-launch` étape `17b` + compteur d'étapes de `skills/saas-factory/SKILL.md` (20 étapes).

20. **(2026-07, archétype automation) L'idempotence a DEUX grains — le châssis ne couvre gratuitement que le mauvais.** Grain **RUN** (`withIdempotency`, « un effet au plus par tick ») ≠ grain **ENTITÉ** (« un manque = au plus UNE entité ouverte »). Toute automation qui **crée des entités** (réappro, relance, ticket, commande) a besoin du grain entité, **entièrement à inventer** (rien à copier) : **clé d'identité déterministe + index unique PARTIEL sur l'état ouvert + upsert RPC atomique** (concurrence-safe, conforme lesson #15). Un worker idempotent au seul grain run **double quand même ses entités** à la première reprise. 🚨 **La clé d'identité EXCLUT tout attribut MUTABLE** (quantité, montant, sévérité) : les inclure re-crée un doublon dès que l'attribut change — la quantité vit dans le PAYLOAD, révisée par l'upsert, jamais dans la clé. Exemple piégeur à bannir : `sha256(sku|besoin|fenêtre)` (`besoin` = quantité mutable). Doctrine : `_shared/archetypes/automation.md §Doctrine AU5`.

21. **(2026-07, archétype automation) La fenêtre d'idempotence run par défaut (24 h + bucket jour) est un piège pour tout worker sub-quotidien.** Un défaut `IDEMPOTENCY_WINDOW_SEC=86400` + bucketisation au jour calendaire **skippe en silence** tous les runs après le 1er du jour : le worker cadencé toutes les N h < 24 h tourne **1×/jour** en croyant tourner toutes les N h, **sans aucune erreur** (l'inverse de l'intention, invisible). **La fenêtre run se DÉRIVE de la cadence** (`≈ SYNC_INTERVAL_HOURS·3600`), jamais d'un défaut 24 h ; tout sub-quotidien DOIT surcharger la clé run + la fenêtre. Motif « fenêtre déterministe » : `bucket = floor((t − EPOCH)/période)`, **EPOCH constante DE CODE** (jamais `now()`/minuit local), **préfixe de clé versionné**.

22. **(2026-07, archétype automation) « BDD optionnelle / fallback fichier » ne vaut QUE sur disque persistant — runner éphémère ⇒ base durable OBLIGATOIRE.** Sur runner éphémère (GitHub Actions `schedule`, Cloud Scheduler, CI, conteneur jetable) le disque est effacé à chaque tick → l'état d'idempotence (`.automation/*.json`) disparaît → l'état repart de zéro → **idempotence cassée** (les deux grains). L'axe « éphémérité » manque partout où « Supabase optionnel » est écrit : le fallback fichier est valide **uniquement** sur disque persistant OU pour un test local one-shot. Cette contrainte **conditionne la reco de déploiement** (éphémère ⇒ Supabase branché avant toute planification réelle).

23. **(2026-07, archétype automation) Un scaffold sans runner de test rend la doctrine TDD inapplicable dès la 1re tâche.** Le pipeline est TDD (« rouge → vert → commit ») ; si le bloc de l'archétype ne livre **ni framework ni script `test`**, l'exécuteur n'a rien pour écrire un rouge — la 1re tâche du build est un vœu. Le scaffold d'un archétype DOIT livrer, dès le départ : framework + script `"test"` (0 dépendance runtime nouvelle — `node:test`+`tsx` suffisent) **+ couture de testabilité** (`fetchImpl` injectable, défaut = `fetch` global ; fixtures/serveur loopback) pour tester une ingestion custom et l'e2e **sans réseau réel**. Preuve : une fois le harness posé, 62 tests écrits sans friction (worker StockSentinel).

24. **(2026-07, archétype automation) La boucle fermée a 3 régimes — « notifier chaque succès » est un anti-pattern (spam).** Notifier chaque run réussi noie l'alerte utile (à N=4 h, ~6 mails/jour). Choisir le régime par **nature d'événement** : (1) **succès silencieux journalisé** (AU2 seulement, défaut d'un run OK) · (2) **rapport périodique planifié** (digest agrégé, cadence propre) · (3) **alerte immédiate** (échec, OU événement métier à forte valeur). Corollaire : un worker réel porte **≥ 2 cadences** → pattern `RUN_MODE` (dispatch par mode, **chaque mode a sa propre fenêtre d'idempotence run**) + helper « domain store » pour l'entité métier (le châssis n'outille nativement que `runs`/`idempotency`). Doctrine : `_shared/archetypes/automation.md §Doctrine AU4`.

25. **(2026-07, transversal) L'archétype-awareness doit vivre dans le CORPS des skills, pas seulement dans `routing.md`.** Un exécuteur fidèle lit la skill + son template, **pas forcément** `routing.md`. Quand seul `routing.md` porte le recadrage (ex. 05/06/10/11 pour automation), l'exécuteur rate les exigences DURES de l'archétype (boucle fermée nommée, idempotence-risque-n°1, scheduler comme host, ROI plein plutôt que canvas). **L'étape 07 prouve que c'est faisable** (baseline de complétude + template + feature-depth conditionnés par archétype) — c'est la cible pour toutes les étapes : le forcing, le contrat d'artefact et le template de chaque skill doivent connaître l'archétype, pas seulement la table de routage.

## Anti-patterns documentés (à prévenir activement)
- **Sur-planification → abandon** : trop de docs avant la première ligne de code.
- **Validation marché tardive** : MVP construit avant tout retour client.
- **Dérive documentaire** : le code change (ex. provider LLM), la doc ne suit pas.
- **Parallélisme non maîtrisé** : bugs à la jonction des features → prévoir une passe d'intégration.

## Profil d'exécution à respecter
Biais pour l'action et la vitesse (fort atout). Failure mode = perfectionnisme menant à l'évitement. Donc : **plafonner la phase de cadrage, montrer du concret vite, décider aux portes, itérer.**
